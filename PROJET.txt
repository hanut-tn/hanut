DESCRIPTION
-----------
Hanut est une plateforme SaaS pour vendeurs tunisiens (e-commerce, Instagram,
WhatsApp). Elle permet de gérer commandes, clients, catalogue, livraisons et
analytiques depuis un tableau de bord web. Les clients passent commande via
un lien public unique (page de commande sans compte).

Stack technique
  - Framework : Next.js 15 (App Router, server components, server actions)
  - Base de données : Supabase (PostgreSQL + RLS + Auth)
  - UI : Tailwind CSS v3
  - Langage : TypeScript strict
  - Monorepo : Turborepo
  - Tests : Vitest (18 tests)
  - Déploiement : Vercel

BASE DE DONNÉES — SCHÉMA SUPABASE
================================================================================

TABLE sellers
  id               UUID PK
  email            TEXT UNIQUE NOT NULL
  name             TEXT NOT NULL
  phone            TEXT
  slug             TEXT UNIQUE            -- lien public boutique
  plan             TEXT DEFAULT 'starter' -- 'starter' | 'pro' | 'business'
  subscription_end TIMESTAMPTZ
  created_at       TIMESTAMPTZ

TABLE products
  id              UUID PK
  seller_id       UUID FK sellers
  name            TEXT
  price           DECIMAL(10,2)
  cost            DECIMAL(10,2)           -- coût d'achat
  stock           INTEGER DEFAULT 0
  low_stock_alert INTEGER DEFAULT 3
  variants        JSONB DEFAULT '[]'      -- tailles, couleurs, etc.
  image_url       TEXT
  created_at      TIMESTAMPTZ

TABLE customers
  id          UUID PK
  seller_id   UUID FK sellers
  name        TEXT
  phone       TEXT
  address     TEXT
  city        TEXT
  tags        TEXT[] DEFAULT '{}'         -- étiquettes libres
  notes       TEXT                        -- notes internes vendeur
  order_count INTEGER DEFAULT 0
  created_at  TIMESTAMPTZ

TABLE orders
  id          UUID PK
  seller_id   UUID FK sellers
  customer_id UUID FK customers
  product_id  UUID FK products
  variant     TEXT
  quantity    INTEGER DEFAULT 1
  cod_amount  DECIMAL(10,2)              -- montant contre-remboursement
  status      TEXT DEFAULT 'new'
              -- 'pending'|'new'|'confirmed'|'shipped'|'delivered'|'returned'
  notes       TEXT
  created_at  TIMESTAMPTZ
  updated_at  TIMESTAMPTZ

TABLE deliveries
  id           UUID PK
  order_id     UUID FK orders
  carrier      TEXT                       -- transporteur
  tracking     TEXT
  fee          DECIMAL(10,2)              -- frais livraison
  cod_collected DECIMAL(10,2)
  cod_reversed  DECIMAL(10,2)
  status       TEXT
  created_at   TIMESTAMPTZ

TABLE team_members                        -- NOUVEAU (étape 6)
  id         UUID PK
  seller_id  UUID FK sellers ON DELETE CASCADE
  user_id    UUID FK auth.users (nullable avant activation)
  role       TEXT CHECK ('admin'|'operator'|'readonly')
  email      TEXT NOT NULL
  name       TEXT
  status     TEXT CHECK ('pending'|'active') DEFAULT 'pending'
  invited_at TIMESTAMPTZ
  joined_at  TIMESTAMPTZ
  UNIQUE (seller_id, user_id)
  UNIQUE (seller_id, email)

TABLE waitlist
  id, email UNIQUE, created_at

TABLE contact_messages
  id, name, email, message, created_at

FONCTIONS SQL (SECURITY DEFINER)
================================================================================

get_seller_id() → UUID
  Retourne le seller_id effectif de l'utilisateur courant :
  - Si vendeur (sellers.id = auth.uid()) → retourne son propre id
  - Si membre d'équipe actif → retourne team_members.seller_id
  Utilisée dans les policies RLS pour l'isolation multi-tenant.

get_team_role(p_seller_id UUID) → TEXT
  Retourne le rôle de auth.uid() vis-à-vis d'un seller :
  - 'admin' si p_seller_id = auth.uid()
  - sinon le rôle dans team_members

can_write_seller(p_seller_id UUID) → BOOLEAN
  Retourne true si le rôle est 'admin' ou 'operator' (pas 'readonly').

create_order_with_stock(...) → UUID
  Fonction atomique (plpgsql) :
  1. Vérifie le stock disponible (SELECT FOR UPDATE)
  2. Crée ou trouve le client par téléphone
  3. Insère la commande
  4. Décrémente le stock
  Utilisée par la page de commande publique et la création interne.

    app/api/team/[memberId]/route.ts  PATCH rôle / DELETE membre
    supabase/migrations/20260602_add_team_members.sql

  Rôles :
    admin     → accès complet, gestion équipe, dashboard, settings
    operator  → commandes + catalogue + livraisons, pas analytics ni settings
    readonly  → consultation uniquement (orders, customers, analytics)

  Sidebar par rôle :
    Dashboard   → admin seulement
    Commandes   → tous
    Clients     → tous
    Catalogue   → admin + operator
    Livraisons  → admin + operator
    Analytiques → admin + readonly
    Paramètres  → admin seulement

  Flux d'invitation :
    1. Admin va dans /settings/team, saisit email + rôle
    2. POST /api/team → insert team_members (status=pending)
    3. supabase.auth.admin.inviteUserByEmail() envoie l'email
    4. L'invité clique le lien, crée son compte Supabase
    5. Au premier chargement du dashboard, layout.tsx active l'invitation
       (UPDATE team_members SET status='active', user_id=user.id)
    6. getUserContext() retourne sellerId = team_members.seller_id

  Limites :
    - Maximum 5 membres par boutique
    - Réservé au plan Business
    - L'admin ne peut pas modifier son propre rôle ni se supprimer
    - readonly bloqué côté serveur sur toutes les écritures

SYSTÈME D'AUTHENTIFICATION ET CONTEXTE
================================================================================

getUserContext() — lib/get-context.ts
  Fonction mise en cache par requête HTTP (React.cache avec fallback React 18).
  Retourne : { userId, sellerId, role, isSeller, plan }

  Logique :
    1. getUser() depuis les cookies de session
    2. Si sellers.id = user.id → role='admin', isSeller=true
    3. Sinon cherche team_members actif → role=member.role, isSeller=false
    4. Retourne null si non authentifié

  Toutes les pages et actions server utilisent context.sellerId
  (jamais user.id directement) pour l'isolation multi-tenant.

middleware.ts
  Protège toutes les routes sauf :
  /, /login, /register, /features, /pricing, /about, /carriers,
  /mobile, /roadmap, /contact, /order/*, /api/contact,
  /api/waitlist, /api/orders/public

  Redirige vers /login si pas de session Supabase.

NOTES IMPORTANTES
================================================================================

1. getUserContext() doit être utilisé dans TOUTES les pages/actions server.
   Ne jamais utiliser user.id directement comme sellerId — les membres
   d'équipe ont un user.id différent du sellerId de leur patron.

2. Le service client (createServiceClient) bypass le RLS. À utiliser
   uniquement quand nécessaire : listes d'équipe, activation invitations,
   lecture du profil vendeur d'un membre.

3. React.cache() est disponible en React 19+. Le fallback
   `reactWithCache.cache ?? ((fn) => fn)` garantit la compatibilité
   avec React 18 et les environnements de test.

4. Les policies RLS sont additives (permissive). Les nouvelles policies
   team_* s'ajoutent aux policies existantes (seller_id = auth.uid()).
   L'accès est accordé si l'une OU l'autre est vraie.

5. git push : la clé SSH n'est pas configurée dans l'environnement Claude.
   Exécuter git push depuis le terminal du projet.
